## Overall Impression

S04 is the anchor screen and the design ambition shows. Three-pane workspace, real interaction patterns (dropdowns, detail slide-in, map pins syncing with list items, modal confirmations), consistent session data, and strong semantic color use for blocks. The map-plus-list pattern is genuinely differentiated for a competitive intel tool.

The biggest gap between the mockup and a production-ready screen is **broken JavaScript on initial load**. There are three real bugs that will throw errors in any browser. Beyond those, the screen's biggest opportunity is **resolving amber's dual meaning** (warning vs. "blocking-you"), **disclosing billing terms in Block modals**, and **adding a data-freshness signal** so operators trust what they're seeing.

The layout and visual treatment are defensible as a starting point for the upgrade. Nothing here needs to be thrown out. Everything below is fixable in a tight pass.

## Critical bugs (code)

These are real defects I found while reading the JS. Any one of them can kill the screen's JavaScript and leave it inert.

### 🔴 P0 — Sparse array in ITEMS (syntax bug)
**Location:** Line 775
**Problem:** The `zen-rival` object closes with `},,` — a double comma that creates a sparse array slot with `undefined`. Any `.filter()` / `.forEach()` / `.find()` loop over `ITEMS` will hit the undefined slot. `renderList()` will crash the first time it tries to read `.type` or `.loc` on the hole.
**Fix:** Change `},,` → `},`

### 🔴 P0 — Stale init code references non-existent IDs
**Location:** Lines 1021–1023
```js
document.getElementById('fcPromo').classList.add('fc-active');
document.getElementById('fcSku').classList.add('fc-active');
document.getElementById('fcBlocked').classList.add('fc-active');
```
**Problem:** These IDs don't exist in the DOM. The current filter UI uses `flt-dd-*` IDs (the dropdown pattern). Calling `.classList` on `null` throws `TypeError: Cannot read properties of null`. This happens before `renderList()` is called on line 1024, so the list never initially populates.
**Fix:** Remove all three lines.

### 🔴 P0 — Stale search selector
**Location:** Line 813
```js
const search = document.querySelector('.search-input').value.toLowerCase();
```
**Problem:** Search input was removed per the comment on line 949 (`/* search removed — Option A */`). `.search-input` does not exist in the DOM. `.value` on `null` throws. `renderList()` can't run.
**Fix:** Either add a no-op fallback (`const search = ''`) or remove the variable entirely and delete line 818's search filter.

> **Verify rendered:** once these three are fixed, open S04 in the browser and confirm the list populates, the map pins render, and the "STIIIZY dropped Blue Dream 22%" alert is selected on load per line 1025.

## CannaSpy-specific checks (7 from the plugin)

### 1. Block-state readability → 🟡 P1

- Rose for "user is blocking rival" → ✅ consistent (STIIIZY, OTC modals; block-status-strip; pin color)
- Amber for "rival is blocking user" → ✅ present (Harborside, Cookies pins with `.pin-blocking-you`)
- **But amber is ALSO used for general warnings** — MedMen promo uses `.chip-warn` with amber background; MedMen pin uses `#c88a20` (amber); `.sev-high` uses amber. Same color, three meanings.

**Impact:** When an operator scans the list and sees an amber chip, they don't know if it means "a rival is blocking me" or "this is a moderate-severity alert." The product's most strategic signal gets diluted.

**Recommendation:** Split the semantic.
- Keep amber (`--warm`) exclusively for "blocking-you."
- Introduce a new token for warning severity — suggest a tempered yellow (`#e8b84a`-ish) or soft orange distinct from amber.
- Update `.chip-warn`, `.icon-warn`, `.sev-high`, `.det-stat-n.cn-warm` to use the new warning token when they mean "warning" (not "blocking-you").

### 2. Platform-name leakage → ✅ Clean
Grepped the full file for Weedmaps / api-g / slug / endpoint / scraping. No hits. "Rival is blocked — they cannot access CannaSpy" is the strongest claim made, which is accurate and safe. The Jungle Boys narrative correctly says "supply chain issue, brand dispute, or compliance hold" without naming a platform.

### 3. Billing action clarity → 🔴 P0

The Block confirmation modals **do not disclose the billing commitment**.

**Current copy (MedMen modal):**
> "Blocking MedMen will add 1 slot to your billing ($200/mo at Elite rate) and prevent them from ever accessing CannaSpy. The block activates immediately."

**What's missing:**
- No mention of auto-renew on the 1st
- No mention of the one-month minimum
- No mention of proration on activation day

**Why it matters:** A customer clicks "Block rival →" thinking they're getting a one-time or flexible commitment, then sees a full $200 charge hit on the 1st of next month and another after that. Billing disputes and trust damage follow.

**Recommended copy:**
> "Blocking MedMen adds 1 slot at $200/mo (Elite rate). Prorated today, full charge on the 1st of each month. One-month minimum. Cancel anytime before the 1st to stop future charges — the slot stays active through month-end. The block activates immediately."

This is longer, but this is where detail earns trust.

### 4. Number typography → ✅ Consistent
Prices (`.price-new`, `.price-old`), stats (`.det-stat-n`), chips (`.li-chip`), timestamps (`.li-time`), and map stat pills are all `var(--mono)` JetBrains Mono with `font-variant-numeric: tabular-nums`. Tabular alignment works. No complaints.

### 5. Block-state differentiation → 🟡 P1
See finding #1 — rose vs. amber is clear, but amber's dual use undermines the signal. Fix by separating warning and blocking-you colors.

### 6. Data freshness → 🟡 P1

Per-alert timestamps (`2h ago`, `5h ago`, `3h ago`) are present and visible ✅. But the **screen has no global freshness signal** — when was the last pipeline scan? When is the next one?

For a daily-cron intel product (2–5 AM Pacific scrape window), operators need this to know if they're looking at 4-hour-old data or 20-hour-old data. A single pill in the top-right corner would do it.

**Recommendation:** Add a freshness pill next to the map-topbar stats.
> "Last scan: today 2:47 AM · Next: tomorrow 2:30 AM"

### 7. Nav active state → 🟡 P1

Command Center is correctly marked `.active` ✅.

**But 6 of the 11 visible nav links are `.locked`**: Alert Feed, Promotions, Block Management, Billing & Slots, Team, and the rest of the Account section. They're greyed out and not clickable.

Per the handoff, all these screens exist as canonical files (S07, S08, S12, S16, S28, S29). This is either leftover "coming soon" styling from an older build or a deliberate choice to hide features — either way it contradicts the product's current state.

**Recommendation:** Remove `.locked` from all 6 links and wire their `onclick` handlers to navigate to the corresponding canonical files. This is a 15-minute fix.

## General design critique

### Layout & hierarchy

**🟡 P1 — Three-pane layout cramps at narrow widths**
Nav (250px) + detail panel when open (380px) + list (360px) + map (flex remainder). At viewport widths under ~1400px, the map becomes too narrow to be useful. No responsive behavior is defined — no breakpoint media queries anywhere in the CSS. MSO users on 13" MacBooks will feel the squeeze.

**🟡 P2 — Location filter and map title disagree**
Location dropdown defaults to "All locations" but the map shows "West Hollywood Flagship" with a pulsing live-dot. The map appears to only ever show one location's neighborhood. Two possibilities:

1. Location filter scopes only the list panel, map is always WeHo (current behavior) — confusing state mismatch
2. Location filter should scope both — map switches to the selected location's view

Option 2 is what most users will expect.

**🟢 P2 — Map as the visual anchor is a bet worth reconsidering**
The map takes visual dominance, but the actionable work happens in the list panel (alerts to review, rivals to inspect, blocks to place). This is a design stance, not a bug — but for a daily-use ops tool, a list-forward layout might convert more work per visit. Worth A/B-testing or at minimum pressure-testing with a real MSO user.

### Interaction patterns

**✅ Detail panel syncs with pin selection** — clicking a list item or a map pin opens the detail panel and highlights the corresponding pin. Good bi-directional sync.

**🟡 P2 — Dropdowns use `onclick` on divs**
Location dropdown (`.loc-dd-opt`) and filter dropdown (`.flt-dd-opt`) are `<div>` elements with `onclick` handlers. No keyboard support, no focus management, not announced as buttons by screen readers. See Accessibility section.

**🟡 P2 — No loading states**
When a user clicks "View price matrix" or "View full profile," the toast says "Opening..." but there's no loading indicator on the current screen. For actions that navigate to heavy data pages, a brief skeleton or progress pill helps.

### Visual polish

**🟢 P3 — Three radial background gradients**
Teal + amber + rose all radiating from corners. For a data-dense screen, the background is doing more work than it needs to. Worth simplifying to one subtle teal glow in the top-left.

**🟢 P3 — Noise texture at 0.022 opacity is imperceptible**
The SVG noise overlay on `body::before` renders at 2.2% opacity in light mode. Adds ~1.5KB to every page and a blend-mode composition layer for no visible effect. Drop it.

**🟢 P3 — Animation timing is scattered**
List hover (0.15s), transitions (0.2s), modal (0.28s), detail panel (0.32s), theme toggle (0.3–0.4s). Consolidate to 2–3 canonical durations (e.g., 0.15s for micro, 0.25s for panels, 0.4s for theme).

## Accessibility (WCAG 2.1 AA)

Enterprise MSO procurement often requires WCAG AA. The current screen fails several criteria.

### 🔴 P0 — Color contrast failures

| Token | On bg | Ratio | Req (AA normal text) | Status |
|---|---|---|---|---|
| `--text-3: #8aafc8` | `--bg: #eef1f6` (light) | ~2.4:1 | 4.5:1 | ❌ Fails |
| `--text-3` (dark) | `--bg: #0c1a1f` | ~4.3:1 | 4.5:1 | ❌ Fails |
| Small labels (9px–10px) | — | — | — | ❌ Below minimum font size |

`--text-3` is used for `.li-time`, `.bss-since`, `.det-section-label`, chip labels, and many secondary elements. Needs to be darkened to ~`#5e89a5` (light) and lightened to `#9ab5c8` (dark) to reach 4.5:1.

### 🔴 P0 — Interactive elements as divs

Locations dropdown options, filter dropdown options, list items, and map pins all use `onclick` handlers on `<div>` elements with no `role`, `tabindex`, or keyboard handlers. Keyboard users cannot navigate S04 at all.

**Fix pattern:** `<div class="list-item" tabindex="0" role="button" onkeydown="handleKey(e,...)">` or convert to `<button>` elements.

### 🟡 P1 — No visible focus styles

No `:focus` or `:focus-visible` rules in the CSS. Keyboard users, when they can navigate, have no visual indication of where they are.

### 🟡 P1 — Icon-only buttons missing labels

- `.zoom-btn` (+ / − map controls) — no aria-label
- `.detail-back` has `title="Back to list"` ✅ but should add `aria-label="Close detail"` for screen readers
- Pin heads (`ST`, `MM`, `OT`, etc.) — decorative initials, but the pin itself is clickable without a descriptive name

### 🟡 P2 — Pin touch targets
Pin heads are 32×32px. WCAG 2.5.5 recommends 44×44px for touch targets. On iPad or 2-in-1 laptops, accurately clicking the correct pin will be frustrating.

## UX copy findings

### 🟡 P1 — Block confirmation modals missing billing terms
Covered in CannaSpy Check #3 above. This is the highest-impact copy fix on the screen.

### 🟡 P1 — "Rival is blocked — they cannot access CannaSpy"
Used in block-status-strip. Works for an investor demo but is abstract for a daily user.

**Before:** "Rival is blocked — they cannot access CannaSpy"
**After:** "You're blocking STIIIZY WeHo. They can't see your pricing."

Names the rival, states what the user is doing, and what the rival can't do — all in one line.

### 🟡 P2 — "Blocking your account — 31 days active"
Used in the Harborside/Cookies list items. "Active" is ambiguous — is the block active, or is the rival active?

**Before:** "Blocking your account — 31 days active"
**After:** "Harborside has been blocking you for 31 days"

### 🟡 P2 — "Enable unblock notification →"
Current CTA on "a rival is blocking you" detail. Passive, process-y.

**Before:** "Enable unblock notification →"
**After:** "Get notified when Harborside drops their block →"

### 🟡 P2 — "12 changes today · 1,700+ SKUs"
Ambiguous. 12 changes in what? Price? Catalog? Promos?

**Before:** "12 changes today · 1,700+ SKUs"
**After:** "12 price or catalog changes today · 1,700+ SKUs"

### 🟢 P3 — "Mark reviewed"
On every alert. Fine, but "Dismiss" is more common and shorter. Consider.

### 🟢 P3 — Empty state
```
No items match current filters.
```
Weak for an intel tool. An empty list is suspicious — help the user recover.

**Before:** "No items match current filters."
**After:** "No alerts match these filters. Today's scan finished at 2:47 AM and flagged 5 items across all locations — clear your filters to see them."

### 🟢 P3 — Map stat pills redundancy
"5 alerts · 7 tracked · 2 blocked" duplicates information visible in the list header. Consider: "5 unreviewed · 7 rivals · 2 active blocks" — more precise.

### ✅ Cancellation warning language is strong
```
Warning: The moment you cancel this block, STIIIZY will be added back to the active prospect list and contacted within 24–48 hours. They will be able to access CannaSpy. This cannot be undone without re-purchasing the block slot.
```
This is exactly the right intensity. Do not soften. The "this cannot be undone without re-purchasing" is especially effective.

## Design system findings

### 🟡 P2 — Hardcoded `#c88a20` amber third shade

The file defines `--warm: #d4900a` (light) and `--warm: #F6C992` (dark). But `#c88a20` appears hardcoded in at least 7 places:

- Line 129: `.sev-high { color: #c88a20 }`
- Line 151: `.det-stat-n.cn-warm { color: #c88a20 }`
- Line 273: `.icon-warn { color: #c88a20 }`
- Line 288: `.chip-warn { color: #c88a20 }`
- Line 539: MedMen pin background
- Line 566: legend swatch for "High alert"

These should all use `var(--warm)`. Maintaining three amber shades manually guarantees drift.

### 🟢 P3 — `--border-2` used inconsistently

Dropdown borders use `--border-2` (22% opacity), but most other cards use `--border` (12% opacity). No clear rule for which to use when. Document or consolidate.

### 🟢 P3 — Repeated inline SVGs

The same chevron-down, map-pin, and filter-bars SVGs appear inline 5–8 times each. Moving to a `<symbol>` + `<use>` pattern or extracting to CSS would reduce file size by ~15% and make icon updates trivial.
